# 形态 Power 蓝图模块架构

## 核心数据流

```text
power JSON
  -> PowerAst
  -> BlueprintGraph
  -> React Flow
  -> BlueprintGraph
  -> PowerAst
  -> power JSON
```

## 核心模型

### PowerAst

表达 JSON 的语义结构，保留：

- `type`
- typed fields
- nested slots
- unknown fields
- source metadata

### BlueprintGraph

表达画布交互结构，包含：

- nodes
- edges
- viewport/layout metadata
- node selection metadata

BlueprintGraph 不应成为最终持久化格式。

React Flow 的 `nodes` / `edges` 只在 `ui/BlueprintCanvas.tsx` 中由 `BlueprintGraph` 临时转换得到，不作为语义源。

## Schema Registry

Schema registry 应能回答：

- 某个 `type` 有哪些字段。
- 字段是普通值还是 slot。
- slot 允许哪些类型。
- 字段是否必填。
- 字段默认值是什么。

## 连接校验

连线必须基于 slot 类型校验。例如：

- entity action 输出只能连接 entity action slot。
- entity condition 输出只能连接 entity condition slot。
- item condition 不能误接 entity condition。

## 测试策略

首批测试应使用现有 power JSON 作为 fixtures：

- 简单 power
- `origins:multiple`
- `origins:action_over_time`
- `shape-shifter-curse:charge_action`

测试重点：

- parse 不报错
- serialize 后语义一致
- graph roundtrip 不丢字段

当前阶段的验证方式：

- `npm run build`
- 后续补充 Vitest 后，增加 `parse -> serialize` 与 `parse -> graph` 的 fixture 测试。
