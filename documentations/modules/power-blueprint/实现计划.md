# 形态 Power 蓝图实现计划

## 目标

实现一个面向 `shape-shifter-curse-fabric` Power JSON 的可视化蓝图编辑器。

第一阶段目标不是完整复刻 UE 蓝图，而是先打通：

```text
Power JSON -> PowerAst -> BlueprintGraph -> 可视化只读图 -> 编辑 -> 保存 BlueprintGraph -> 导出 Power JSON
```

## 核心判断

### 顶层 Power 导出规则

如果一个 Power JSON 文件只包含单个 power，则导出时不应额外包一层 `origins:multiple`。

只有当一个 Power JSON 文件需要承载多个命名 power 时，才使用：

```json
{
  "type": "origins:multiple",
  "power_name": {
    "type": "..."
  }
}
```

因此：

- 单 power 蓝图的根节点就是该 power 本身。
- multi power 蓝图的根节点才是 `origins:multiple`。
- 从 Graph 导出 JSON 时，不能为了统一模型强行包 `origins:multiple`。
- 解析已有 JSON 时应保留原始顶层结构类型。

### 节点粒度

应生成节点的对象：

- power 节点
- entity action 节点
- entity condition 节点
- item action 节点
- item condition 节点
- bientity action 节点
- 复杂 datatype 节点，例如 `Vector`、`Particle Effect`

留在节点内部参数块的对象：

- string
- number
- boolean
- enum
- identifier
- 简单 list
- 其他无独立连线价值的标量字段

### 数据源

Apoli 固定实现：

- 从 `F:\MC Modding\RefMods\apoli` 整理为固定 schema。
- 不依赖在线文档作为最终事实来源。
- 可先手写覆盖高频类型，后续再批量补齐。

SSC 自定义实现：

- 来源于 `shape-shifter-curse-fabric/src/main/java/net/onixary/shapeShifterCurseFabric/additional_power`。
- 应建立源码扫描/半自动 schema 生成工具。
- 解析目标是 `PowerFactory`、`ActionFactory`、`ConditionFactory` 和 `SerializableData.add(...)`。
- 生成结果进入可编辑的本地 schema 文件，允许人工修正特殊类型。

## 数据模型修正

当前 `PowerAst` 草案需要补充：

- 字段顺序
- 数组 slot 的稳定顺序
- optional field 显示状态
- unknown field 保留
- datatype node 支持
- `origins:multiple` 的命名子 power 支持
- 单 power 顶层直接导出，不自动包 `origins:multiple`

建议模型核心字段：

```ts
type SlotKind =
  | "power"
  | "entity_action"
  | "entity_condition"
  | "item_action"
  | "item_condition"
  | "bientity_action"
  | "block_action"
  | "block_condition"
  | "damage_condition"
  | "fluid_condition"
  | "biome_condition"
  | "datatype";

interface AstNode {
  id: string;
  type: string;
  kind: SlotKind;
  label?: string;
  fields: AstField[];
  unknownFields: AstUnknownField[];
}

interface AstField {
  name: string;
  value: AstValue;
  schema?: FieldSchema;
  visible: boolean;
}
```

## Schema Registry

每个 schema 至少需要描述：

- `type`
- `kind`
- 显示名称
- 节点颜色
- 字段列表
- 字段类型
- 是否必填
- 默认值
- 是否可多连线
- 是否允许 `inverted`

slot 字段示例：

```ts
{
  name: "actions",
  valueKind: "slot_array",
  slotKind: "entity_action",
  required: true,
  multi: true
}
```

普通字段示例：

```ts
{
  name: "volume",
  valueKind: "number",
  required: false,
  defaultValue: 1.0
}
```

datatype 字段示例：

```ts
{
  name: "spread",
  valueKind: "datatype",
  datatype: "vector",
  required: false
}
```

## 连线规则

基础规则：

- 输出端口类型必须与输入 slot 类型兼容。
- 一个子节点的输入端口只接受一条来自父参数端口的连接。
- 非数组参数端口只能连接出一条线，对应一个子对象。
- 参数类型为 Array 时，该参数端口可以连接出多条线，每条线对应一个 Array 成员。
- Array 参数端口的多条输出连接需要保存顺序。
- condition 节点统一支持 `inverted` 参数。
- datatype 节点只能连接到对应 datatype 字段。

连接方向：

- 当前设计稿以父节点字段作为输出端口，连到子节点输入端口。
- 数据语义上仍表示父 JSON 持有子 JSON。
- 不设计“多条输出线连接到同一个输入节点”的情况。

## 自动布局规则

首次从 JSON 生成蓝图时：

1. 生成根 power 节点。
2. 按字段顺序生成 slot 子节点。
3. 从左到右排列层级。
4. 同一数组内从上到下排列。
5. datatype 节点靠近使用它的字段右侧。
6. 默认视口聚焦根 power 节点。

布局保存后优先读取 `.ssc-studio` 蓝图存档，不再强制重排。

## 阶段计划

### 阶段 1：示例 Power 只读生成

目标：

- 只支持 `form_axolotl_fountain_belt_trinket.json` 所需类型。
- 实现 JSON -> AST -> Graph。
- 在当前 Power 蓝图页面展示静态可视化节点和连线。

需要覆盖的类型：

- `origins:multiple`
- `shape-shifter-curse:action_on_sprinting_to_sneaking`
- `shape-shifter-curse:falling_protection`
- `origins:and`
- `origins:add_velocity`
- `origins:play_sound`
- `shape-shifter-curse:explosion_damage_entity`
- `origins:gain_air`
- `origins:spawn_particles`
- `origins:target_condition`
- `origins:in_tag`
- `origins:owner`
- `origins:on_block`
- `origins:air`
- `Vector`
- `Particle Effect`

验收：

- 选择示例 Power 后能生成与设计稿接近的只读蓝图。
- 节点颜色、端口位置、参数块基本符合设计稿。
- 默认视口聚焦根 power 节点。

### 阶段 2：保存 Studio 蓝图数据

目标：

- 支持拖动节点。
- 支持保存/读取节点位置。
- 支持保存孤立节点。
- 支持保存视口。

验收：

- 关闭再打开同一 Power，节点位置保持。
- 没有连接到根节点的孤立节点不丢失。

### 阶段 3：Graph -> AST -> JSON 导出

目标：

- 从蓝图重新生成 Power JSON。
- 导出前验证必须存在 power 节点。
- 保留 unknown fields。
- 单 power 图导出为单个 power 对象。
- 多 power 图才导出为 `origins:multiple`。

验收：

- 示例 Power roundtrip 后语义一致。
- 非法图保存时给出提示，不写入 JSON。
- 对普通单 power 文件 roundtrip 后不会被额外包成 `origins:multiple`。

### 阶段 4：节点创建与连接

目标：

- 从输出端口拖线到空白区域时弹出创建菜单。
- 菜单按 slot 类型过滤。
- 创建后自动连接。
- 支持空格/双击空白创建节点。

验收：

- entity action slot 只展示 entity action。
- entity condition slot 只展示 entity condition。
- 非数组字段拒绝第二条连接。

### 阶段 5：可选字段与编辑器

目标：

- 新节点只展示必填字段。
- 双击节点打开可选字段选择弹窗。
- 支持编辑标量字段。
- 支持 condition 的 `inverted`。

验收：

- 未显示的可选字段按默认值导出。
- 显示字段编辑后能正确导出 JSON。

### 阶段 6：schema 扩展

目标：

- 建立 Apoli 固定 schema 数据集。
- 建立 SSC 自定义 schema 生成器。
- 将现有 449 个 Power JSON 纳入兼容性测试。

验收：

- 大部分现有 Power 能打开。
- 未识别 type 会生成 unknown 节点，但不丢字段。
