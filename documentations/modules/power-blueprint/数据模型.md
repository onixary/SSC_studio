# 形态 Power 蓝图数据模型

## 原则

Power JSON 是外部持久化格式，PowerAst 是内部语义格式，BlueprintGraph 是 UI 交互格式。

不要把 React Flow nodes/edges 直接保存为 power 文件。

如果一个 JSON 文件只包含单个 power，顶层必须直接导出该 power；只有一个文件包含多个命名 power 时，才使用 `origins:multiple` 作为顶层容器。内部 AST 可以统一表达根节点，但序列化时必须保留这种语义差异。

`origins:multiple` 只是同一 JSON 中承载多个命名 power 的容器，不应作为可见蓝图节点。Graph 投影应跳过该容器，直接显示其中的每个子 power；从 Graph 导出时，如果同一蓝图包含多个 power，再自动补回 `origins:multiple`。

## 当前 PowerAst 初始实现

```ts
type SlotKind =
  | "power"
  | "entity_action"
  | "entity_condition"
  | "item_action"
  | "item_condition"
  | "bientity_action"
  | "bientity_condition"
  | "block_action"
  | "block_condition"
  | "damage_condition"
  | "fluid_condition"
  | "biome_condition"
  | "datatype"
  | "unknown";

interface AstNode {
  id: string;
  type: string;
  kind: SlotKind;
  label?: string;
  fields: AstField[];
  unknownFields: AstUnknownField[];
}

interface AstField {
  name: string;
  value: AstValue;
  schema: FieldSchema;
  visible: boolean;
}

type AstValue =
  | string
  | number
  | boolean
  | null
  | AstNode
  | AstNode[]
  | Record<string, unknown>;
```

## 当前 BlueprintGraph 初始实现

```ts
interface BlueprintGraph {
  nodes: BlueprintNode[];
  edges: BlueprintEdge[];
}

interface BlueprintNode {
  id: string;
  astNodeId: string;
  type: string;
  kind: SlotKind;
  title: string;
  color: string;
  position: { x: number; y: number };
  fields: BlueprintGraphField[];
}

interface BlueprintGraphField {
  name: string;
  schema: FieldSchema;
  valueKind: FieldValueKind;
  displayValue: string;
  connected: boolean;
  color: "power" | "action" | "condition" | "datatype" | "scalar" | "unknown";
  inputKind?: "int" | "float" | "string" | "boolean";
  sourceHandleId: string;
}

interface BlueprintEdge {
  id: string;
  source: string;
  sourceHandle: string;
  target: string;
  targetHandle: string;
  fieldName: string;
  color: string;
  order?: number;
}
```

数组字段的多条连线共享同一个字段输出端口，并通过 `order` 保存数组成员顺序。

当前 UI 中已经允许临时修改 React Flow 投影：

- 节点位置可拖动调整，并会在保存时写入 `.ssc-studio/blueprints/<power>.json`。
- 新建节点、连线、可选字段增删、选择删除目前会进入基础 `BlueprintGraph -> PowerAst -> JSON` 保存链路。
- 标量参数编辑目前通过 `BlueprintGraphField.displayValue` 投影，保存时按 `inputKind` 转回 JSON 标量。
- slot/datatype 参数块的 `connected` 与连接态 `displayValue` 在当前 UI 中由 React Flow edges 推导；它们是画布投影状态，不应作为独立语义源保存。
- 选择状态使用 React Flow 的 `selected` 投影状态，删除节点时会同步移除相连边。
- 框选使用 partial selection，节点与框选区域局部重叠即会被选中。
- `origins:multiple` 不生成可见节点，多个子 power 会作为多个根 power 节点显示。
- 这些交互结果目前已进入基础 `BlueprintGraph -> PowerAst -> JSON` 导出链路，后续仍需要补齐更完整的 schema 校验和 roundtrip 测试。
- Power 节点的 `label/title` 可在节点头部编辑。单 power 导出时该名称只保存在 `.ssc-studio` UI 状态中；多 power 根节点导出为 `origins:multiple` 时，该名称作为 multiple 容器中的字段名。

## 蓝图 UI 状态

`.ssc-studio/blueprints/<power>.json` 是 Studio 自有 UI 状态文件，不是 Power JSON 的语义源。当前保存：

- 节点 id、astNodeId、类型签名与位置。
- React Flow 视口。
- 当前 BlueprintGraph 快照，用于调试和后续兼容迁移。
- 单 power 蓝图中 Power 节点的内部名称。

打开 Power 时会优先按节点 id / astNodeId 恢复位置；如果 id 因 JSON 重建变化而不匹配，则按 `kind/type/label/title` 签名和出现序号尝试恢复。

Power 的语义内容仍以 `powers/<power>.json` 为准，`.ssc-studio/blueprints` 不直接决定导出的 Power 结构。

## 参数输入

字段 schema 可通过 `numberKind` 区分 `int` 和 `float`。Graph 投影中使用 `inputKind` 描述节点内输入框行为：

- `int`：只接受整数，小数点不合法。
- `float`：接受整数或小数，提交时按浮点数规范化显示。
- `string`：直接接收输入。
- `boolean`：使用节点内开关控件提交 `true` / `false`，不再通过文本输入。

文本输入提交发生在回车或输入框失焦时；非法输入不更新当前字段值。过长文本参数通过输入框和参数块整体向右延伸展示，不自动换行。

## 上下文 Schema

同一个 `type` 可能在不同 registry 中表示不同 kind，例如 Apoli 的 `origins:and`：

- `action/meta/AndAction.createFactory()` 只声明 `actions`。
- `condition/meta/AndCondition.createFactory()` 只声明 `conditions`。

因此 schema registry 需要按 `type + kind` 建立上下文 schema，而不能只用 `type` 作为唯一键。否则 action meta 会错误出现 condition 参数，condition meta 也会错误出现 action 参数。

当前节点库已经为常见 Apoli meta action / meta condition 按 kind 分别注册，例如：

- `origins:and` / `origins:or` action 节点按当前 action kind 暴露 `actions`。
- `origins:and` / `origins:or` condition 节点按当前 condition kind 暴露 `conditions`。
- `origins:chance` 在 action 上下文中暴露 `action` / `fail_action` / `chance`，在 condition 上下文中只暴露 `chance`。

对于暂未精确注册的节点，`registry.inferFieldSchema()` 会按字段名推断 `entity_action`、`bientity_action`、`item_action`、`block_action` 以及各类 condition slot，作为保底策略。该推断只用于避免未知节点丢失基本连接语义，不能替代正式 schema。

## 未知字段策略

解析时：

- schema 已知字段进入 typed fields。
- schema 未知字段进入 `unknownFields`。
- 当前阶段为了不丢字段，未识别但可推断为标量、slot、datatype 的字段也会进入 `fields`，并携带推断 schema。

保存时：

- typed fields 按 schema 输出。
- unknown fields 合并回原始对象。
- datatype `vector` 导出为 `{ x, y, z }`，不会额外写入内部 `type: "vector"`。
- datatype `particle_effect` 如果原始语义是简单字符串，则导出回字符串。
- 其他对象型 datatype 节点在 AST 中使用内部 `type` 标识节点种类，但导出时会去掉内部 `type`，只输出该 data type 的字段对象。

## Datatype 表示

- `vector` 在 AST 中使用内部节点 `type: "vector"`，UI 显示为 `Vector`。
- `particle_effect` 在 AST 中固定使用内部节点 `type: "particle_effect"`，UI 显示为 `Particle Effect`。
- Particle Effect 的实际粒子标识符保存在字段 `type` 中，`params` 作为普通 string 字段显示。
- 简单字符串粒子在导出时仍可回写为字符串，带 `params` 的粒子导出为对象。
- 当前已按本地 Apoli 2.9.2 源码确认并注册一批对象型 datatype 节点，包括 `attribute_modifier`、`attributed_attribute_modifier`、`damage_source_description`、`food_component`、`hud_render`、`ingredient`、`item_stack`、`key`、`positioned_item_stack`、`status_effect_instance`、`text`。
- 枚举、标识符、字符串、数字、布尔等简单 data type 仍作为标量字段处理，不单独注册为蓝图节点。

## 2026-06-08 Datatype array 更新

- `FieldValueKind` 已新增 `datatype_array`，用于表示参数本身是某个 data type 的数组。
- `datatype_array` 在 Graph 中与 `slot_array` 一样使用同一个字段输出端口的一对多连线表示，每条连线对应一个数组成员，`BlueprintGraphEdge.order` 保存导出顺序。
- `datatype_array` 的连线颜色和节点类型按 `datatype` 主题处理。
- 字符串/枚举包装型 data type 现在也会注册为可创建节点，例如 `Identifier`、`Comparison`、`Space`、`NBT`、`Item Slot` 等。节点内部使用一个标量输入字段编辑，导出 JSON 时会回到对应的 primitive 值，数组字段会回到 primitive 数组。
- 对象型 data type 继续导出为不包含 Studio 内部 `type` 字段的对象；`vector` 和 `particle_effect` 保留各自的特殊序列化规则。
- Apoli 的通用 `Modifier.DATA_TYPE` 与 Minecraft 属性用 `Attribute Modifier` 分开建模。`Modifier.LIST_TYPE` 对应字段使用 `datatype_array`，包括 `modifiers`、`food_modifiers`、`saturation_modifiers`、`delta_modifiers`、`hardness_modifiers` 等。
- 如果解析到数组字段但 schema 仍无法判断数组成员类型，该字段会进入 `unknownFields` 并随原 JSON 保留，不再投影为不可编辑的 `Array[n]` 参数块。
